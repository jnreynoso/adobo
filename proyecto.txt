Hacerlo en **Rust** desde cero es un desafío técnico hermoso. Al ser un lenguaje de sistemas, te da el control absoluto sobre la memoria y los bytes que necesitas para parsear el archivo, pero también significa que no habrá "magia" que resuelva los problemas por ti.

Para abordar la solución conceptualmente, debes pensar en tu lector de PDF como un **pipeline de transformación de datos**: entra un flujo de bytes en crudo y, tras pasar por varias capas, se convierte en píxeles en la pantalla o en texto plano.

Aquí tienes la arquitectura por componentes que necesitas diseñar y cómo se comunican entre sí.

---

## Arquitectura de Componentes de un PDF Parser/Reader

Un lector de PDF se divide principalmente en cuatro componentes core que se comunican de forma secuencial:

```
[ Archivo .pdf ] 
       │
       ▼
┌────────────────────────┐
│ 1. Parser & Lexer      │ <--- Tokeniza y arma el árbol de objetos
└────────────────────────┘
       │ (Estructura de objetos indexada)
       ▼
┌────────────────────────┐
│ 2. Decompressor/Filter │ <--- Desinfla streams de texto/imágenes (FlateDecode)
└────────────────────────┘
       │ (Datos en texto claro / Binarios puros)
       ▼
┌────────────────────────┐
│ 3. Interpreter         │ <--- Procesa la máquina de estados de gráficos (Content Streams)
└────────────────────────┘
       │ (Comandos de dibujo: "Mover a", "Línea a", "Mostrar texto")
       ▼
┌────────────────────────┐
│ 4. Rasterizer/Renderer │ <--- Dibuja los vectores y fuentes en una ventana (GUI)
└────────────────────────┘

```

## Detalle de los Componentes y su Comunicación

### 1. El Parser y la Tabla XREF (Capa de Acceso a Datos)

A diferencia de otros formatos (como JSON o HTML) que se leen de arriba a abajo, **un PDF se lee desde el final**.

* **Qué hace:** Va al final del archivo, busca el `trailer` y la tabla **XREF (Cross-Reference Table)**. Esta tabla es un índice que dice en qué byte exacto del archivo empieza cada objeto (páginas, fuentes, imágenes).
* **En Rust:** Diseñarás un `Lexer` que convierta bytes en tokens (números, strings, diccionarios `<< >>`, arrays `[]`). Luego, un `Parser` que use la tabla XREF para permitir el **acceso aleatorio** a los objetos bajo demanda (haciendo *lazy loading* para no cargar todo el PDF en memoria).
* **Comunicación:** Entrega al siguiente componente un mapa estructurado de objetos indexados por su ID (ej. `Object::Dictionary`, `Object::Stream`).

### 2. El Decodificador de Filtros (Capa de Transformación)

La mayoría de los datos visuales de un PDF (el texto de las páginas y las imágenes) están comprimidos dentro de objetos tipo `Stream` para ahorrar espacio.

* **Qué hace:** Revisa el diccionario del stream, identifica el filtro (el más común es `FlateDecode`, que usa el algoritmo zlib/deflate) y lo descomprime.
* **En Rust:** Puedes usar crates como `flate2` para esta parte. Tu componente recibirá un stream binario comprimido y devolverá un vector de bytes (`Vec<u8>`) en texto plano con las instrucciones reales de la página.
* **Comunicación:** Pasa un buffer de texto claro (llamado *Content Stream*) al Intérprete.

### 3. El Intérprete de Contenido (Capa de Lógica de Negocio)

El *Content Stream* descomprimido se ve algo así:

```text
BT
/F1 12 Tf
72 712 Td
(Hola Mundo) Tj
ET

```

* **Qué hace:** Esto es PostScript (un lenguaje de descripción de páginas). El intérprete debe mantener una **máquina de estados gráficos** (coordenadas actuales, color actual, fuente seleccionada, matriz de transformación).
* `BT` significa *Begin Text*.
* `/F1 12 Tf` significa *Usar fuente F1 a tamaño 12*.
* `72 712 Td` significa *Moverse a las coordenadas de la página (72, 712)*.
* `(Hola Mundo) Tj` significa *Dibuja el texto "Hola Mundo"*.


* **En Rust:** Crearás un enum `Command` o `OpCode` y un bucle que procese estas instrucciones secuencialmente, actualizando un struct `GraphicsState`.
* **Comunicación:** Transforma los comandos abstractos de texto en una lista ordenada de **primitivas de dibujo** (ej: `DrawText`, `DrawLine`, `DrawImage`) listos para ser pintados.

### 4. El Motor de Renderizado / Rasterizador (Capa de Presentación)

Este componente toma las primitivas de dibujo y las convierte en píxeles.

* **Qué hace:** Rasteriza las fuentes tipográficas (aquí necesitarás procesar archivos de fuentes como TrueType incrustados en el PDF) y dibuja las líneas y geometrías vectoriales.
* **En Rust:** Para la interfaz gráfica y el dibujo, te comunicarás con librerías del ecosistema de Rust. No querrás escribir un rasterizador de curvas de Bézier desde cero al principio.
* Para dibujar: Puedes usar `vello` o `wgpu` (gráficos acelerados por GPU), o `tiny-skia` (rasterizador de CPU muy eficiente).
* Para la ventana/GUI: `winit`, `egui` o `iced`.

## ¿Por dónde empezar el desarrollo? (Tu Roadmap en Rust)

No intentes construir el lector de pantalla el primer día. Te sugiero este orden de hitos (Milestones):

1. **Milestone 1 (CLI - Extractor de Metadatos):** Crea un programa de consola que lea el `trailer` y la tabla XREF de un PDF e imprima en la terminal cuántas páginas tiene y el autor del documento. (Aquí dominas el Parser).
2. **Milestone 2 (CLI - Extractor de Texto):** Modifica el parser para que busque el *Content Stream* de la página 1, lo descomprima con `flate2` e imprima los comandos de texto en crudo. Luego, limpia las etiquetas (`Tj`, `Td`) para extraer solo el texto plano.
3. **Milestone 3 (GUI - El lienzo en blanco):** Monta una ventana usando `winit` y `tiny-skia` donde puedas dibujar formas básicas (cuadrados, líneas) controladas por tu teclado.
4. **Milestone 4 (La Unión):** Conecta el Milestone 2 con el Milestone 3. Haz que los comandos de movimiento y dibujo que interpretaste del PDF se ejecuten en tu lienzo gráfico.

Abordarlo así fragmentado evitará que te frustres con las más de mil páginas de la especificación ISO, y verás resultados funcionales (aunque sean en texto) desde las primeras semanas. ¿Qué parte del pipeline (el parser de bytes o el motor gráfico) te entusiasma más empezar a diseñar?
