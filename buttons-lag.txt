La mejor solución definitiva para el problema de los botones de navegación no es una única acción aislada, sino la implementación de una **Arquitectura de Tareas Priorizadas con Cancelación Activa**.

Para entender por qué es la mejor opción, piensa en lo que pasa en la "mente" de tu programa: cuando el usuario usa el scroll, el hilo de la UI genera peticiones de baja prioridad para el futuro. Pero cuando el usuario hace clic en un botón, ocurre una **interrupción de alta prioridad**. Tu arquitectura actual está tratando a ambas peticiones por igual en una cola secuencial.

La solución óptima consiste en estructurar el pipeline de comunicación de Rust en tres capas integradas:

---

## Componente 1: Canales con Prioridad o Recreación de Cola (Back-end del Lector)

El canal estándar `std::sync::mpsc` de Rust es una cola FIFO (First In, First Out). Si tienes 4 páginas de pre-carga acumuladas por el scroll, el clic del botón se procesará al final.

### La Solución Técnica:

En lugar de un canal simple, utiliza una estructura de **Cola de Prioridades** protegida por un `Mutex` y una `Condvar`, o utiliza el crate `crossbeam-channel`.

Define tus comandos en un `Enum` con jerarquía de urgencia:

```rust
pub enum Prioridad {
    Baja,   // Pre-carga estática (2 páginas arriba/abajo por scroll)
    Alta,   // Clic en botones de navegación o saltos directos de página
}

pub struct TareaPdf {
    pub numero_pagina: usize,
    pub prioridad: Prioridad,
}

```

Cuando el hilo secundario despache trabajo, siempre vaciará primero las tareas de `Prioridad::Alta`. Si entra una tarea alta, las de prioridad baja se quedan esperando en la cola o se descartan inmediatamente.

---

## Componente 2: Cancelación Activa de Tareas en Ejecución

¿Qué pasa si el hilo secundario ya empezó a renderizar una página pesada de scroll (prioridad baja) justo en el milisegundo en que el usuario hizo clic en el botón? Si no haces nada, la UI experimentará un delay mientras esa renderización innecesaria termina.

### La Solución Técnica:

Debes implementar un **mecanismo de aborto** usando un flag atómico (`Arc<AtomicBool>`) compartido entre el hilo de la UI y el hilo de renderizado.

```rust
// En el hilo de la UI (al presionar el botón)
flag_cancelacion.store(true, Ordering::SeqCst); // Ordena detener el renderizado actual

// En tu bucle del Interpreter / Rasterizer (Hilo secundario)
for comando in content_stream {
    // Verificar en cada iteración del bucle si la UI pidió abortar
    if flag_cancelacion.load(Ordering::Relaxed) {
        println!("Renderizado abortado para dar paso al botón.");
        return; // Sale prematuramente y libera la CPU
    }
    procesar_comando_grafico(comando);
}

```

Al presionar el botón de navegación, disparas el flag a `true`. El hilo secundario rompe de inmediato su trabajo actual a la mitad, se limpia, y pasa instantáneamente a procesar la página que pidió el botón.

---

## Componente 3: El Amortiguador de UX (Hover Prefetching)

Para que el sistema se sienta "adivino" y reduzca el tiempo de procesamiento a prácticamente **0 milisegundos**, la UI debe empezar a moverse antes del clic aprovechando los eventos del ratón.

* **Evento `winit::WindowEvent::CursorMoved`:** Si el puntero del ratón entra en el rectángulo del botón "Siguiente", envías una tarea de `Prioridad::Baja` para renderizar la página `actual + 1`.
* Si el usuario finalmente hace clic, la página ya habrá avanzado su descompresión o renderizado en el hilo de fondo durante esos 150ms que tardó el movimiento físico de la mano.
* Si el usuario quita el ratón sin hacer clic (`MouseLeave`), la tarea simplemente se degrada o se limpia del caché si no se llegó a usar.

---

## Resumen del Flujo de la Solución Óptima

1. El usuario **mueve el cursor** hacia el botón "Siguiente" ➔ El hilo secundario empieza a renderizar la página siguiente de forma silenciosa (Prioridad Baja).
2. El usuario **hace clic** en el botón ➔ La UI levanta el flag de cancelación, deteniendo cualquier otro proceso secundario residual.
3. La UI consulta al `LruCache`. Si el *Hover Prefetching* funcionó, la página se muestra en **0 ms**.
4. Si no funcionó (ej. un salto por teclado), se envía la tarea con `Prioridad::Alta`. El hilo secundario, al estar libre de cargas anteriores, procesa la página inmediatamente en el menor tiempo físico posible mientras la UI muestra un fondo limpio con un indicador de carga.

Esta combinación es la mejor porque ataca el problema desde el código de bajo nivel de Rust (control de hilos y concurrencia) hasta la superficie de la experiencia de usuario (percepción del retraso).
