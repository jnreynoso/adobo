Lo que estás experimentando es el comportamiento por defecto de un sistema *Lazy* puro: como solo mantiene en memoria lo que se ve en pantalla (más un pequeño margen), en cuanto una página sale del área de visión, el recolector la destruye para liberar RAM. Al volver a subir, el sistema se ve obligado a reconstruirla desde cero, causando ese molesto re-procesamiento.

Para que las páginas no se vuelvan a cargar al subir y bajar, necesitas complementar tu estrategia de *Lazy Rendering* con un **Mecanismo de Caché de Páginas (LRU Cache)**.

La regla de oro aquí es: **Lazy para la carga inicial, Caché para la persistencia temporal.**

---

## La Solución: Implementar una Caché LRU (Least Recently Used)

En Rust, no quieres guardar *todas* las páginas renderizadas para siempre porque un PDF de 500 páginas agotaría la memoria RAM de inmediato. Necesitas una estructura de datos que mantenga un número máximo de páginas pre-renderizadas (por ejemplo, 5 o 10 páginas) y que, cuando se llene, elimine automáticamente la página que lleva más tiempo sin ser vista.

### 1. Estructura Conceptual del Caché

En lugar de que tu visor le pida la página directamente al renderizador, ahora se la pedirá al componente de Caché:

```
[Usuario hace Scroll]
       │
       ▼
¿La página está en el Caché?
 ├── SÍ ──> [Retornar Buffer de Píxeles Inmediatamente] (0 ms de delay)
 └── NO ──> [Renderizar en Hilo Secundario] ──> [Guardar en Caché] ──> [Mostrar]

```

### 2. Implementación en Rust

Para implementar una caché LRU eficiente en Rust sin reinventar la rueda, puedes usar crates probados como `lru` o `cached`. Si prefieres mantener las dependencias al mínimo, puedes simular una usando una combinación de `HashMap` y un `VecDeque` (para el orden de prioridad), o simplemente el crate `lru::LruCache`.

Aquí tienes un ejemplo conceptual de cómo estructurar este componente en tu Milestone 4:

```rust
use lru::LruCache;
use std::num::NonZeroUsize;

// Representa los píxeles ya renderizados por tiny-skia
pub struct RenderedPage {
    pub width: u32,
    pub height: u32,
    pub pixels: Vec<u8>, // Tu buffer de pixeles RGBA
}

pub struct PdfViewerCache {
    // Almacena un máximo de N páginas en memoria
    storage: LruCache<usize, RenderedPage>, 
}

impl PdfViewerCache {
    pub fn new(capacity: usize) -> Self {
        Self {
            storage: LruCache::new(NonZeroUsize::new(capacity).unwrap()),
        }
    }

    pub fn get_or_render<F>(&mut self, page_number: usize, render_fn: F) -> &RenderedPage
    where
        F: FnOnce() -> RenderedPage,
    {
        // Si la página ya existe, la devuelve y la marca como "recientemente usada"
        // Si no existe, ejecuta el renderizador, la guarda en el caché y la devuelve
        self.storage.get_or_insert(page_number, || {
            println!("Caché MISS para la página {}. Renderizando...", page_number);
            render_fn()
        })
    }
}

```

### 3. Cómo integrarlo en tu ciclo de renderizado (`winit`)

En tu bucle de eventos principal, cuando determines qué páginas intersectan con la pantalla actual (por ejemplo, las páginas 1 y 2), harás la llamada a través del caché:

```rust
// Supongamos que el usuario está viendo la página 2
let posicion_scroll_pagina = 2;

// El caché intentará recuperar la página. 
// Al volver a subir, como la página 1 ya está en el almacenamiento interno de la LRU,
// no ejecutará la pesada función de renderizado de fondo.
let pagina_visible = cache.get_or_render(posicion_scroll_pagina, || {
    // Tu lógica actual del Milestone 2 y 4:
    // 1. Ir al stream del PDF
    // 2. Descomprimir e interpretar
    // 3. Rasterizar con tiny-skia
    motor_pdf.renderizar_pagina_completa(posicion_scroll_pagina)
});

// Dibujar el buffer 'pagina_visible.pixels' en la ventana
canvas.draw_bitmap(&pagina_visible.pixels, ...);

```

## Ajustes de rendimiento para este enfoque

* **Capacidad del Caché:** Para reportes estándar, una capacidad de **4 a 6 páginas** suele ser el punto dulce. Te permite tener en memoria la página actual, la anterior, la siguiente, y un margen para cuando el usuario hace scroll rápido hacia arriba o hacia abajo.
* **Caché en segundo plano (Prefetching):** Si el usuario está quieto en la página 2, puedes aprovechar los ciclos ociosos de la CPU para pedirle al caché de forma asíncrona la página 3 y la página 1. Así, cuando decida moverse, el caché dará un *Hit* instantáneo.
* **Liberación de memoria:** Al usar `LruCache`, cuando el usuario llegue a la página 10, las páginas 1 y 2 se desalojarán (`drop`) automáticamente de la memoria, manteniendo el consumo de RAM de tu lector completamente controlado y constante.
